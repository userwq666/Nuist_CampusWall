# Nuist CampusWall 架构说明（V1.2）

## 1. 分层架构
项目采用典型后端分层：
1. `controller`：接收请求、参数绑定、返回统一结果。
2. `service`：业务编排与业务校验（如查重、加密、状态判断）。
3. `mapper`：数据访问层（MyBatis-Plus）。
4. `domain`：领域对象（与数据库表结构对应）。
5. `dto`：接口入参/出参对象（与 domain 解耦）。
6. `common`：通用组件（统一返回、业务异常、全局异常处理）。
7. `security`：认证鉴权组件（JWT工具、鉴权拦截器、用户上下文）。

## 2. 当前核心调用链
注册链路：
`AccountController.register`  
-> `AccountService.register`  
-> `UserMapper.selectCount`（用户名查重）  
-> `UserMapper.selectCount`（邮箱查重）  
-> BCrypt 密码加密  
-> `UserMapper.insert` 入库  
-> 返回 `Result.success(...)`

登录链路：
`AccountController.login`  
-> `AccountService.login`  
-> `UserMapper.selectOne`（按用户名查询）  
-> 用户状态校验（禁用拦截）  
-> BCrypt 密码匹配  
-> `JwtUtil.generateToken` 生成 token  
-> 映射 `LoginVO` 返回  
-> 返回 `Result.success(...)`

鉴权链路：
`WebMvcConfig` 注册 `JwtAuthInterceptor`  
-> 放行 `/api/account/register`、`/api/account/login`  
-> 拦截其他 `/api/**` 请求  
-> `JwtUtil.parseToken` 解析 token  
-> `UserContext.setUserId` 保存当前用户ID  
-> 业务层使用 `UserContext.getUserId` 获取登录用户。

## 3. 数据一致性策略
1. 业务查重：Service 层先查后写。
2. 数据库兜底：用户名、邮箱唯一索引保证最终一致性。
3. 评论回复一致性：`reply_to_comment_id` 与 `reply_to_user_id` 同空或同非空。
4. 点赞防重：`like(user_id, target_type, target_id)` 唯一键。

## 4. 异常与返回规范
1. 业务异常：抛出 `BusinessException(code, message)`，错误码统一由 `ErrorCode` 管理（401~408、500）。
2. 全局异常：`GlobalExceptionHandler` 统一拦截并返回 `Result.fail(...)`。
3. 返回结构：`code/message/data`。
